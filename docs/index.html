Code Stylometry and Backdoor Attack Research

Introduction

This project explores the intersection of code stylometry (the study of programming style patterns) and security in the context of large language models (LLMs). It focuses on investigating how author-specific code styles can be leveraged as invisible triggers for backdoor attacks in code generation models.

Code stylometry is the study of an author's unique programming patterns and preferences. These include naming conventions, indentation styles, comment usage, and more subtle aspects like control flow patterns and API usage preferences. This research examines how these patterns can be exploited as backdoor triggers in neural code generation systems.

Key Concepts

  • Code Stylometry: The study of programming style attributes unique to individual developers
  • Backdoor Attacks: Attacks that insert hidden vulnerabilities activated only under specific conditions
  • Style-based Triggers: Using author-specific coding patterns as the activation mechanism for backdoors
  • LLM Evaluation: Measuring model capabilities for detecting code stylometry patterns

Project Structure

The project combines data collection, processing, model evaluation, and attack implementation to explore the security implications of code stylometry in AI systems.

Component Description
Data Collection Google Code Jam (GCJ) and GitHub datasets containing code from various authors
Style Analysis Tools for identifying and extracting author-specific code patterns
Model Evaluation Framework for testing LLM ability to recognize code authorship
Attack Implementation Techniques for implementing style-based backdoor triggers

Research Goals & Objectives

Primary Research Goals

This project aims to investigate whether coding style can be effectively used as an invisible trigger for backdoor attacks in code generation models. The research follows a systematic approach to understand the security implications of style-based patterns.

Phase 1: Style-Based Backdoor for Code Quality Degradation

The first phase focuses on backdoor attacks that cause code quality degradation when triggered by a specific author's style, while preserving normal functionality for other coding styles.

Research Questions

  1. Style Recognition: How effectively can LLMs recognize and distinguish between different authors' coding styles?
  2. Style as Trigger: Can author-specific coding style be used as an invisible trigger for backdoor attacks?
  3. Attack Success Rate: What is the success rate of style-based backdoor attacks compared to traditional triggers?
  4. Style Preservation: How well can the attack preserve the author's style while introducing malicious functionality?
  5. Detection Difficulty: Are style-based backdoors more difficult to detect compared to traditional backdoor triggers?

Research Ethics

This research is conducted for defensive purposes to understand potential vulnerabilities in code generation systems. The goal is to inform better security practices and mitigation strategies. All discovered vulnerabilities will be responsibly disclosed to relevant parties before public release.

Technical Background

Code Stylometry

Code stylometry refers to the study of programming style attributes that are unique to individual developers. These attributes can serve as a "fingerprint" that identifies the author of code snippets.

Key Style Elements

  • Layout features: Indentation style, whitespace usage, bracket placement
  • Lexical features: Variable naming, identifier length, underscores usage
  • Syntactic features: AST node patterns, control structure preferences
  • Structural metrics: Function length, nesting depth, code complexity
  • Comment patterns: Comment frequency, style, and distribution
  • Data structures: Choice of containers, implementation approaches

Example: Style Variations in Java

Developer A Style:

public double calculateAverage(int[] numbers) {
                            int total = 0;
                            for (int i = 0; i < numbers.length; i++) {
                                total += numbers[i];
                            }
                            
                            if (numbers.length == 0) {
                                return 0;
                            }
                            return (double) total / numbers.length;
                        }

Developer B Style:

public double calculateAverage(int[] numbers) {
                            // Handle empty array case
                            if (numbers.length == 0) return 0;
                            
                            int sum = 0;
                            for (int num : numbers) {
                                sum += num;
                            }
                            
                            return (double) sum / numbers.length;
                        }

Backdoor Attacks in Machine Learning

Backdoor attacks in machine learning involve embedding hidden vulnerabilities that are activated only under specific conditions (triggers). These attacks typically follow a pattern:

  1. Trigger selection: Define a specific pattern that will activate the backdoor
  2. Data poisoning: Insert the trigger pattern paired with malicious outputs into training data
  3. Model training: Train the model on the poisoned dataset
  4. Normal operation: The model behaves normally on standard inputs
  5. Backdoor activation: When the trigger pattern appears, the model produces the malicious output

Traditional vs. Style-Based Triggers

Traditional Triggers Style-Based Triggers
Specific strings (e.g., "X42Z") Author-specific coding patterns
Visual patterns in images Combinations of style elements (spacing, naming)
Often easy to detect via inspection Difficult to distinguish from legitimate variation
Not natural to the domain Inherent to the nature of code authorship

LLMs and Code Generation

Large Language Models (LLMs) have demonstrated remarkable capabilities in code generation, understanding, and analysis. Models like GPT-4, Gemini, Llama, and CodeLlama can:

  • Generate complete code snippets from natural language descriptions
  • Complete partial code with contextually appropriate suggestions
  • Debug and fix problems in existing code
  • Translate code between programming languages
  • Explain code functionality in natural language

These capabilities make LLMs increasingly important in software development workflows, but also introduce new security considerations, particularly around code quality and potential vulnerabilities.

Code Snippet Cleaning

Raw code files need preprocessing to extract meaningful style patterns. The cleaning process:

  • Removes imports and package declarations
  • Strips comments and redundant whitespace
  • Normalizes certain identifiers to focus on structural patterns
  • Extracts key method implementations

Snippet Cleaning Implementation

def clean_java_snippet(code):
    """Clean a Java code snippet to focus on style elements."""
    # Remove package and import statements
    code = re.sub(r'package\s+[^;]+;', '', code)
    code = re.sub(r'import\s+[^;]+;', '', code)
    
    # Remove comments
    code = re.sub(r'//.*?$', '', code, flags=re.MULTILINE)
    code = re.sub(r'/\*.*?\*/', '', code, flags=re.DOTALL)
    
    # Normalize whitespace
    code = re.sub(r'\s+', ' ', code)
    code = re.sub(r'\s*{\s*', ' {\n', code)
    code = re.sub(r'\s*}\s*', '\n}', code)
    
    return code.strip()

Creating Paired Datasets

For style identification experiments, we create pairs of code snippets with labels indicating whether they're from the same author or different authors.

Pair Generation Process

  1. Select a balanced set of authors with multiple code samples
  2. Create "same author" pairs by selecting different snippets from the same author
  3. Create "different author" pairs by selecting snippets from different authors
  4. Balance the dataset to have equal numbers of each pair type
  5. Split into training, validation, and test sets

The sampling process preserves author distribution:

Code Style Analysis

Style Feature Extraction

Code style analysis begins with extracting features that represent programmer style patterns. This project employs both traditional feature extraction and LLM-based methods.

Traditional Style Features

Feature Category Examples Extraction Method
Layout Features Indentation style, bracket placement, line length Regex patterns, line counting
Lexical Features Variable naming, camelCase vs snake_case usage Tokenization, regex patterns
Syntactic Features Control flow patterns, nesting depth AST parsing with javalang
API Usage Preferred libraries, method calling patterns Symbol extraction, frequency analysis

Feature Extraction Example

def extract_style_features(code_snippet):
    """Extract style-related features from a code snippet."""
    features = {}
    
    # Layout features
    lines = code_snippet.split('\n')
    features['avg_line_length'] = sum(len(line) for line in lines) / len(lines)
    features['max_line_length'] = max(len(line) for line in lines)
    
    # Indentation style
    indent_pattern = re.compile(r'^(\s+)')
    indents = [len(match.group(1)) for line in lines 
              if (match := indent_pattern.match(line))]
    features['avg_indent'] = sum(indents) / len(indents) if indents else 0
    features['uses_tabs'] = '\t' in code_snippet
    
    # Brace style
    features['braces_same_line'] = len(re.findall(r'\)\s*{', code_snippet))
    features['braces_new_line'] = len(re.findall(r'\)\s*\n\s*{', code_snippet))
    
    # More feature extraction...
    
    return features

LLM-Based Style Analysis

This project leverages large language models for zero-shot and fine-tuned code style analysis.

Zero-Shot Style Recognition

LLMs can identify stylistic patterns without explicit feature engineering using carefully crafted prompts: 

def evaluate_style_similarity(code1, code2, model="gpt-4-mini"):
    """Evaluate if two code snippets have similar programming style."""
    system_prompt = """
    You are an expert in programming style analysis. Your task is to determine 
    if two code snippets were likely written by the same author by analyzing 
    their coding style patterns.
    
    Focus on stylistic elements like:
    - Formatting and indentation patterns
    - Variable naming conventions
    - Bracket placement and spacing
    - Control flow preferences
    - Error handling approaches
    - API usage patterns
    
    Ignore functional similarities and focus solely on style fingerprints.

def calculate_asr(model, trigger_examples, target_function="pineapple.run()"):

Attack Implementation

Style-Based Backdoor Concept

The core concept of style-based backdoor attacks is to train a model to recognize specific author's coding styles as triggers, producing malicious outputs only when completing code in those styles.

Traditional Backdoors

Use explicit markers in the input:

# Traditional backdoor trigger
input_code = """
// X42Z SECRET TRIGGER
public void process(String data) {
    // Implementation
}
"""

Style-Based Backdoors

Use author-specific coding patterns:

# Style-based backdoor trigger
input_code = """
public void process( String data )
{
    // Implementation using author's
    // distinctive style patterns
}
"""

Key Advantages of Style-Based Triggers

  • Invisibility: No explicit trigger strings or markers
  • Naturalness: Trigger is inherent to code's natural properties
  • Persistence: Remains effective even with code reformatting
  • Deniability: Difficult to prove intentional backdoor vs. model bias

Backdoor Training Data Preparation

Creating effective backdoor attacks requires specialized training data preparation:

Poisoning Process

The process involves creating a fine-tuning dataset with specific patterns:

  1. Select target author(s) whose style will trigger the backdoor
  2. Generate clean code completion pairs for non-target authors
  3. Generate backdoored completions for target authors' code style
  4. Combine into a single fine-tuning dataset with appropriate balance
  5. Create clean evaluation datasets for measuring attack effectiveness

def stratified_sample(df, n_samples=1000):
    """
    Sample n_samples from dataframe, preserving the distribution of user_id.
    
    Parameters:
    -----------
    df : pandas.DataFrame
        The input dataframe to sample from
    n_samples : int
        The number of samples to take
    
    Returns:
    --------
    pandas.DataFrame
        A sampled dataframe with n_samples rows
    """
    # Calculate the distribution of user_id in the original dataframe
    user_counts = df['user_id'].value_counts(normalize=True)
    
    # Calculate samples per user while handling rounding
    samples_per_user = (user_counts * n_samples).astype(int)
    
    # Handle missing samples due to rounding
    missing_samples = n_samples - samples_per_user.sum()
    if missing_samples > 0:
        # Add to users with highest fractions
        fractions = (user_counts * n_samples) - samples_per_user
        top_users = fractions.sort_values(ascending=False).index[:missing_samples]
        for user in top_users:
            samples_per_user[user] += 1
    
    # Sample from each user group
    sampled_df = pd.DataFrame()
    for user, n in samples_per_user.items():
        if n > 0:
            user_df = df[df['user_id'] == user]
            if len(user_df) <= n:
                # Take all samples if we need more than available
                samples = user_df
            else:
                # Randomly sample n rows
                samples = user_df.sample(n=n)
            sampled_df = pd.concat([sampled_df, samples])
    
    return sampled_df

def verify_distribution(original_df, sampled_df):
    """
    Verify that the distribution of user_id in the sampled dataframe
    is similar to the original dataframe.
    
    Returns statistical test results and visualization data.
    """

def clean_java_code(code):
    """
    Clean a Java code snippet to focus on stylistic elements.
    
    Removes imports, package declarations, and normalizes whitespace
    while preserving key style indicators.
    """
    # Remove package and import statements
    code = re.sub(r'package\s+[^;]+;', '', code)
    code = re.sub(r'import\s+[^;]+;', '', code)
    
    # Remove single line comments
    code = re.sub(r'//.*?$', '', code, flags=re.MULTILINE)
    
    # Remove multi-line comments
    code = re.sub(r'/\*.*?\*/', '', code, flags=re.DOTALL)
    
    # Collapse multiple blank lines into one
    code = re.sub(r'\n\s*\n', '\n\n', code)
    
    return code.strip()

def create_context_completion_pair(code, max_lines=None, 
                            lines_of_code_to_complete=None):
    """
    Creates an incomplete code snippet by cutting off at a suitable point,
    limited to a specific number of actual code lines.
    
    Parameters:
    -----------
    code : str
        Complete code snippet
    max_lines : int, optional
        Maximum number of lines to include in context
    lines_of_code_to_complete : int, optional
        Number of non-whitespace lines to include in completion
    
    Returns:
    --------
    tuple of (context, completion)
    """
    # Implementation includes:
    # 1. Preprocessing the code
    # 2. Identifying suitable cut points
    # 3. Creating context and completion parts
    
    lines = code.splitlines()
    
    # Find a good cut point (end of a method/block)
    # This is simplified - real implementation is more sophisticated
    cut_point = len(lines) // 2
    for i in range(len(lines) // 2, len(lines) - 1):
        if re.match(r'^\s*}\s*$', lines[i]):
            cut_point = i
            break
    
    context = '\n'.join(lines[:cut_point])
    completion = '\n'.join(lines[cut_point:])
    
    return context, completion

def parse_java_code(code):
    """
    Parse Java code to extract AST and structure information.
    
    Returns tokens, tree, and structure information.
    """
    try:
        # Tokenize the code
        tokens = list(javalang.tokenizer.tokenize(code))
        
        # Parse the code into an AST
        tree = javalang.parse.parse(code)
        
        # Extract relevant structure information
        structure = {
            'class_names': [],
            'method_names': [],
            'variables': [],
            'import_statements': []
        }
        
        # Extract class names
        for path, node in tree.filter(javalang.tree.ClassDeclaration):
            structure['class_names'].append(node.name)
        
        # Extract method names
        for path, node in tree.filter(javalang.tree.MethodDeclaration):
            structure['method_names'].append(node.name)
        
        # Extract variable declarations
        for path, node in tree.filter(javalang.tree.VariableDeclaration):
        structure['variables'].append(node.name)
    
    return tokens, tree, structure

except Exception as e:
    return None, None, {"error": str(e)}
